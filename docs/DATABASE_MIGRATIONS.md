# Database Migrations

## Purpose

AI Logix uses Alembic to manage database schema changes. Migrations replace unmanaged schema evolution and reduce the risk of inconsistent PostgreSQL state across environments.

## Location

```text
backend/alembic.ini
backend/alembic/env.py
backend/alembic/script.py.mako
backend/alembic/versions/
```

## Configuration

Alembic reads the database URL from `app.core.config.settings.database_url`. Do not hardcode credentials in Alembic files.

`backend/alembic/env.py` imports all current SQLAlchemy models so `Base.metadata` is complete for autogenerate.

## Docker Compose Commands

Run these from the repository root after the backend container is running:

```bash
docker compose exec backend alembic -c alembic.ini current
docker compose exec backend alembic -c alembic.ini history
docker compose exec backend alembic -c alembic.ini upgrade head
docker compose exec backend alembic -c alembic.ini downgrade -1
docker compose exec backend alembic -c alembic.ini revision --autogenerate -m "describe change"
```

The backend container mounts `./backend` at `/app`, so `alembic.ini` is available at `/app/alembic.ini`.

## Local Non-Docker Commands

If dependencies are installed locally:

```bash
cd backend
alembic -c alembic.ini current
alembic -c alembic.ini history
alembic -c alembic.ini upgrade head
alembic -c alembic.ini downgrade -1
alembic -c alembic.ini revision --autogenerate -m "describe change"
```

## Creating A New Migration

1. Modify SQLAlchemy models.
2. Generate a migration:

   ```bash
   docker compose exec backend alembic -c alembic.ini revision --autogenerate -m "describe change"
   ```

3. Review the generated migration manually.
4. Confirm it does not drop or alter data unexpectedly.
5. Run tests and apply locally:

   ```bash
   docker compose exec backend alembic -c alembic.ini upgrade head
   ```

## Applying Migrations

```bash
docker compose exec backend alembic -c alembic.ini upgrade head
```

Apply migrations before starting application code in non-development deployments.

## Checking Status

```bash
docker compose exec backend alembic -c alembic.ini current
docker compose exec backend alembic -c alembic.ini history
```

## Downgrade

```bash
docker compose exec backend alembic -c alembic.ini downgrade -1
```

Downgrade support is basic for the baseline migration. Future migrations that transform or delete data must document downgrade limitations.

## Manual Review Policy

- Review every autogenerated migration before applying it.
- Do not trust autogenerate blindly.
- Check table names, column types, nullability, indexes, unique constraints, and foreign keys.
- Confirm data-destructive operations are intentional and backed up.

## Applied Migration Policy

- Do not edit migrations already applied to shared environments.
- Create a new revision to fix or evolve schema.
- Keep migration history linear unless a documented branching strategy is approved.

## Backup Policy

Before production migrations:

- Take a database backup.
- Confirm restore procedure.
- Review migration SQL.
- Define rollback plan.
- Avoid destructive changes without explicit approval.

## Existing Database Baseline Adoption

Some local databases were created before Alembic through development-only `Base.metadata.create_all`. In those databases, running the baseline migration can fail with duplicate-table errors because the schema already exists but the `alembic_version` table does not.

Use this process for an existing database:

1. Compare the live schema against `backend/alembic/versions/20260517_0001_baseline_schema.py`.
2. Confirm the expected tables, columns, indexes, and foreign keys match.
3. If the schema matches, mark the baseline as already applied:

   ```bash
   docker compose exec backend alembic -c alembic.ini stamp head
   ```

4. Confirm status:

   ```bash
   docker compose exec backend alembic -c alembic.ini current
   ```

Do not run `stamp head` blindly. Stamping tells Alembic to trust that the database already matches the revision.

For a fresh local database, the preferred flow is:

```bash
docker compose up -d db
docker compose run --rm backend alembic -c alembic.ini upgrade head
docker compose up --build
```

Development startup still allows `create_all` temporarily for local compatibility. Non-development environments must apply migrations before app startup.
